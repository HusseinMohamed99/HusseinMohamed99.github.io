Case Study · Real Estate · Saudi Market

SystemPro

Overview

SystemPro is a real estate platform built for the Saudi market, letting buyers and renters search properties with smart filters, bookmark listings, and keep browsing even without a connection. It was one of my earlier deep dives into performance work — profiling the app under real usage rather than guessing at bottlenecks.

OfflineFull property browsing via Hive cache
BLoCPagination + dependency injection throughout
Core Capabilities

What the app does

🔍
Property search & filters

Location, price range, and property-type filters that compose without needing a full re-fetch for every change.

🔖
Bookmarks

Saved listings persist locally and sync back once the user is online again.

📡
Real-time updates

New listings and price changes propagate to the browsing list without a manual refresh.

📴
Offline access

Hive-backed local caching means a spotty connection in the field doesn't stop a broker from showing listings.

Architecture

BLoC with pagination and DI baked in

The listing feed uses cursor-based pagination through BLoC, backed by dependency injection so repositories can be swapped for tests without touching UI code. Hive stores the last-known property set locally, which is what keeps search usable offline. The 30% performance gain came from profiling widget rebuilds and network calls under real scrolling behavior, not from a single fix — mostly trimming unnecessary rebuilds in the list and batching image loads.

UI (BLoC)→Repository (DI)→Dio (remote)→Hive (offline cache)

Build failure
A missing .env file silently broke the build

Recently resolved a build failure traced back to an empty .env file — the app compiled but crashed on launch because required config values were never injected. Added a startup check that fails loudly instead of shipping a silently broken build.
